**저장 구조**
- `dashboard/main` — 현재 화면(항상 최신)

**Firestore 규칙 변경 불필요** — 기존 `dashboard/{docId}` 규칙이 날짜 문서까지 커버.

## 6. 사진을 Firebase Storage로 분리 — ❌ 폐기됨 (사진 기능 자체 제거)

> **최종 결정:** Storage 비용(Blaze) 부담으로 이 안은 채택하지 않고, **사진 기능을 앱에서 완전히 제거**했습니다.
> 아래 내용은 당시 검토 기록으로만 남깁니다. 현재 앱에는 사진 관련 코드/화면/스타일이 없습니다.

**무엇을 (폐기된 안)** — 사진을 base64로 문서에 통째로 넣던 방식 → **Storage에 파일 업로드 후 문서엔 URL만 저장**.

**왜 (당신이 지적한 문제)**
- Firestore 문서 1MiB 한계에 빨리 도달
- 사진이 늘수록 문서가 커져 열람이 느려짐
- (추가로 발견) 이력 기능이 매 저장마다 사진까지 문서에 복제 → 문제 가중

**바뀐 점**
- 문서엔 짧은 URL만 저장 → 문서 경량화, 이력 문서도 사진 복제 안 함
- 사진 품질 상향: 최대 1600px, 약 500KB (기존 1280px/170KB)
- 기존 base64 사진도 그대로 표시됨(**하위 호환**). 다시 올리면 URL 방식으로 교체.
- Storage 미설정 시 "Storage가 아직 설정되지 않았습니다" 안내

**⚠️ 당신이 콘솔에서 해야 할 설정 (설정 안 하면 사진 업로드 실패)**

1. Firebase 콘솔 → **빌드 → Storage → 시작하기**
   - 최근 프로젝트는 **Blaze(종량제) 업그레이드**를 요구할 수 있음
   - Blaze도 매월 무료 제공량(저장 5GB / 하루 다운로드 1GB)이 커서 이 사용량은 **사실상 무료**, 단 카드 등록 필요
   - 위치: `asia-northeast3 (서울)` 권장
2. **규칙(Rules)** 탭 → 아래로 교체 후 게시
   ```
   rules_version = '2';
   service firebase.storage {
     match /b/{bucket}/o {
       match /photos/{allPaths=**} {
         allow read: if true;                    // 누구나 사진 열람
         allow write: if request.auth != null;   // 로그인한 시공사만 업로드
       }
     }
   }
   ```

**검토 포인트 / 트레이드오프**
- **Blaze 업그레이드가 부담이면** 이 변경은 보류하고 예전 base64 방식으로 되돌릴 수 있음(사진 몇 장 수준이면 base64도 당장은 동작). 결정은 당신 몫.
- 사진 교체/삭제 시 **Storage의 옛 파일은 자동 삭제하지 않음**(이력 문서가 옛 URL을 참조할 수 있어 일부러 남김). 장기적으로 미사용 파일이 쌓일 수 있으나 무료 제공량 내라 실사용 문제는 거의 없음.

## 7. 저장 충돌 방지

**무엇을** — 저장을 **트랜잭션**으로 감싸고, 편집 시작 시점의 `updatedAt`과 현재 문서를 비교해 그 사이 다른 저장이 있었으면 덮어쓰지 않고 중단.

**왜 (당신이 지적한 문제)** — 저장이 전체 덮어쓰기라, 시공사 계정을 두 기기에서 동시에 쓰면 나중 저장이 먼저 저장을 조용히 날림.

**동작** — 충돌 감지 시: 저장 취소 → "다른 사람이 먼저 저장했습니다. 최신 내용으로 새로고침했으니 다시 편집" 안내 → 최신 화면 표시.

**검토 포인트** — "마지막에 편집한 사람이 이긴다"가 아니라 "먼저 저장한 사람이 이기고, 늦은 사람은 다시 편집"하는 방식. 단일 시공사 운영이면 충돌 자체가 드물지만, 조용한 데이터 유실은 확실히 막음.

## 8. 시간 기준 KST 고정

**무엇을** — 날짜/시간 계산·표시를 전부 `Asia/Seoul`(KST)로 고정.

**대상 함수** — `todayStr`(작업일자·이력 날짜 id), `fmtDate`(요일 표시), `ddayInfo`(D-day), `fmtUpdated`(최종 갱신 시각).

**왜** — 기존엔 브라우저 로컬 시간 기준이라, 기기 시간대가 다르면 작업일자·이력 날짜·D-day가 하루 어긋날 수 있었음.

**검토 포인트** — 이력 문서 id(날짜)도 KST 기준으로 결정됨. 자정 경계가 한국시간 기준으로 딱 맞음.
